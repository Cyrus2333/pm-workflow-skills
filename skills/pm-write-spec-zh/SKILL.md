# PM Write Spec ZH

这个 skill 用于把已澄清的需求输入整理成正式方案。

输出目标：

- 业务、产品、研发都能快速看懂
- 结构完整但不过度堆砌
- 明确范围、边界、风险与验收口径

## 优先读取

默认先读：

1. [PRD 模板](./references/PRD模板.md)
2. [PRD 示例](./references/PRD示例.md)

若当前在具体项目仓库内，再读：

1. `AGENTS.md`
2. 根 `README.md`
3. 项目文档索引
4. 与需求直接相关的规则、页面、运营、实现文档

## 什么时候使用

当用户出现以下表达时优先使用：

- “帮我写需求文档”
- “整理成 PRD”
- “出一版方案说明”
- “给研发评审用”
- “把刚才讨论内容整理成正式文档”

## 编写原则

- 先写背景与问题，再写方案
- 先写业务变化，再决定是否补实现细节
- 范围与非范围必须同时出现
- 状态、权限、异常分支不能省略
- 如果仍有未拍板决策，要单独列出，不要埋进正文

## 工作流程

### 1. 先判断输入是否足够

如果以下信息仍缺失，不要硬写完整 PRD：

- 目标角色不清
- 页面 / 模块不清
- 当前问题不清
- 关键状态分支不清
- 非目标不清

缺失时，先回退到 `pm-requirement-intake-zh`。

### 2. 组织方案结构

默认使用以下结构：

### 1. 需求摘要

### 2. 背景与问题

### 3. 目标与非目标

### 4. 用户与场景

### 5. 方案范围

### 6. 方案说明

### 7. 规则与边界

### 8. 联动影响

### 9. 验收标准

### 10. 待确认问题

### 3. 控制实现细节粒度

默认只写到“研发可评审”的粒度，不默认下钻：

- 接口字段
- 数据库表结构
- 代码改法

只有用户明确要求或上下文需要时，才补这些内容。

### 4. 写清楚风险

如果需求涉及：

- 旧规则迁移
- 历史数据兼容
- 跨页面联动
- 多角色差异
- 审核与消息联动

要单独列风险与验证建议。

## 输出标准

- 让非研发也能读懂
- 让研发能快速识别工作量与风险
- 避免“优化体验”“支持一下”这类空泛描述
- 尽量把每个变化写成“现状 vs 目标”

## 与其他 skills 的衔接

- 需求未澄清时，先用 `pm-requirement-intake-zh`
- 还在发散方案时，先用 `pm-brainstorm-zh`
- 文档形成后，向团队同步时转 `pm-stakeholder-update-zh`
