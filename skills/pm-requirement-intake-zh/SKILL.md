# PM Requirement Intake ZH

这个 skill 用于“提需求前置收口”。

目标不是直接写实现方案，而是把模糊需求变成清晰、可讨论、可进一步写 PRD 的结构化输入。

## 优先读取

默认先读本 skill 自带资料：

1. [提需模板](./references/提需模板.md)
2. [提需示例](./references/提需示例.md)
3. [需求边界检查清单](./references/需求边界检查清单.md)

如果当前在具体项目仓库内，再补读：

1. `AGENTS.md`
2. 根 `README.md`
3. 项目文档索引
4. 与本次需求最相关的产品 / 规则 / 运营文档

## 什么时候使用

当用户出现以下表达时优先触发：

- “这里想优化一下”
- “这个页面要改一下”
- “后台想补个能力”
- “这块逻辑不太顺”
- “帮我梳理需求”
- “想提给研发，但还不够清楚”

## 核心原则

不要接受模糊提法直接进入实现。

至少先明确：

1. 改的是哪个页面、模块、流程或规则域
2. 给哪个角色使用
3. 当前现状是什么
4. 期望变化是什么
5. 哪些状态要单独处理
6. 哪些权限或规则边界不能漏
7. 哪些内容明确不在本次范围内

## 工作流程

### 1. 判断需求类型

先判断它更接近哪类：

- 页面体验优化
- 流程改造
- 规则变更
- 权限调整
- 后台能力补充
- 数据口径调整

如果跨多类，直接指出，不要伪装成单点改动。

### 2. 定位对象

把需求落到具体对象：

- 页面
- 模块
- 入口
- 流程节点
- 规则域

如果用户说不清，主动缩小候选范围。

### 3. 强制补齐字段

至少补齐以下字段：

| 字段 | 说明 |
|------|------|
| 需求标题 | 一句话概括 |
| 目标角色 | 谁使用 |
| 入口 | 从哪里进入 |
| 页面 / 模块 | 改哪里 |
| 当前问题 | 现在哪里不合理 |
| 期望变化 | 改完后要怎样 |
| 主动作 | 用户最核心动作 |
| 次动作 | 次级动作或兜底动作 |
| 关键状态 | 哪些状态要区分 |
| 权限边界 | 谁能看、谁能做 |
| 联动影响 | 审核、消息、搜索、导出、埋点、后台等 |
| 非目标 | 本次明确不做什么 |
| 验收信号 | 怎么算改对了 |

### 4. 识别策略问题

以下情况要标记为“策略待定”，不要假装已经能开发：

- 是否改产品定位
- 是否调整角色体系
- 是否改变核心排序或分发规则
- 是否统一跨场景规则口径
- 是否引入新的激励、等级、评价机制

### 5. 产出结构化需求输入

输出建议结构：

### 1. 需求摘要

### 2. 目标范围

### 3. 当前现状

### 4. 期望变化

### 5. 状态与权限边界

### 6. 可能联动范围

### 7. 明确非目标

### 8. 待确认问题

### 9. 供研发阅读的简版需求描述

## 与其他 skills 的衔接

- 还在发散阶段时，先用 `pm-brainstorm-zh`
- 需求边界清楚后，转 `pm-write-spec-zh`
